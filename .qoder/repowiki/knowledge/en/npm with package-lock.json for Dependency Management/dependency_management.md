## Dependency Management System

This repository uses **npm** as its primary package manager with a `package-lock.json` file (lockfileVersion 3) to ensure deterministic dependency resolution across environments.

### Package Manager Configuration

- **Primary tool**: npm (evidenced by `package-lock.json` presence)
- **Lockfile version**: 3 (modern npm format, compatible with npm 7+)
- **Registry**: Public npm registry (`https://registry.npmjs.org/`) â€” all dependencies resolve from the default public registry
- **No private registries**: No `.npmrc`, no custom registry configuration detected
- **No vendoring**: Dependencies are installed to `node_modules/` (standard npm behavior), not vendored into source control

### Key Dependency Files

| File | Purpose |
|------|---------|
| `package.json` | Declares direct dependencies and devDependencies with semantic versioning ranges |
| `package-lock.json` | Locks exact versions of entire dependency tree (7418 lines, ~700+ transitive dependencies) |
| `.gitignore` | Excludes `node_modules/`, includes selective Yarn artifacts (`.yarn/patches`, `.yarn/plugins`, etc.) suggesting multi-package-manager compatibility |

### Version Pinning Strategy

The project uses a **mixed versioning approach**:
- **Exact pinning** for core framework packages: `next`, `react`, `react-dom`, `eslint-config-next` use exact versions (e.g., `"16.2.9"`)
- **Caret ranges** (`^`) for most other dependencies, allowing minor/patch updates (e.g., `"^10.7.7"`)
- This balances stability for critical framework packages with flexibility for utility libraries

### Multi-Package-Manager Compatibility

The `.gitignore` file reveals intentional support for multiple package managers:
- Ignores `node_modules/` (npm standard)
- Ignores `.pnp.*` files (Yarn Plug'n'Play)
- Selectively tracks `.yarn/patches`, `.yarn/plugins`, `.yarn/releases`, `.yarn/versions` (Yarn Berry features)
- Ignores debug logs for npm, yarn, and pnpm

The README documents installation commands for npm, yarn, pnpm, and bun, indicating the project is designed to work with any of these tools, though `package-lock.json` confirms npm as the primary/authoritative choice.

### Module Resolution

TypeScript configuration uses `"moduleResolution": "bundler"` which aligns with modern bundler-based resolution (Next.js built-in), avoiding Node.js-style resolution quirks. Path aliases are configured via `"@/*": ["./src/*"]` in `tsconfig.json`.
